1 # 2-D Decomposition Overview
2 
3 **Author:** George Vandenberghe
4 
5 **Date:** March 2022
6 
7 The 1D decomposition can read state from a model forecast file, either by reading on rank 0
8 and scattering, or by doing MPI_IO on the model history file using either nemsio, sigio,
9 or netcdf serial or parallel I/O. Very old
10 post tags also implement the more primitive full state broadcast or (a
11 performance bug rectified 10/17) read the entire state on all tasks.
12 This is mentioned in case a very old tag is encountered. The 2D decomposition
13 only supports MPI_IO for the general 2D case but all I/O methods remain
14 supported for the 1D special case of the 2D code. This 1D special case
15 works for all cases currently supported by older 1D tags and branches.
16 
17 to repeat, ONLY 2D NETCDF PARALLEL I/O WILL BE SUPPORTED FOR THE
18 GENERAL CASE OF 2D DECOMPOSITION.
19 
20 **************************** 2D design enhancements ************************
21 
22  The 2D decomposition operates on subdomains with some latitudes and
23  some longitudes. The subdomains are lonlat rectangles rather than
24  strips. This means state must be chopped into pieces in any
25  scatter operation and the pieces reassembled in any gather
26  operation that requires a continuous in memory state. I/O and halo
27  exchanges both require significantly more bookkeeping.
28 
29 
30  The structural changes needed for the 2D decomposition are
31  implemented in MPI_FIRST.f and CTLBLK.f! CTLBLK.f contains
32  numerous additional variables describing left and right domain
33  boundaries. Many additional changes are also implemented in EXCH.f
34  to support 2D halos. Many additional routines required addition of
35  the longitude subdomain limits but changes to the layouts are
36  handled in CTLBLK.f and the "many additional routines" do not
37  require additional changes when subdomain shapes are changed and
38  have not been a trouble point.
39 
40 
41  Both MPI_FIRST and EXCH.f contain significant additional test code
42  to exchange arrays containing grid coordinates and ensure EXACT
43  matches for all exchanges before the domain exchanges are
44  performed. This is intended to trap errors in the larger variety of
45  2D decomposition layouts that are possible and most of it can
46  eventually be removed or made conditional at build and run time.

49 The following is found in CTLBLK.f and shared in the rest of UPP
50 through use of CTLBLK.mod
51 
52  im integer full longitude domain
53  jm integer full latitude domain
54 
55  jsta integer start latitude on a task subdomain
56  jend integer end latitude on a task subdomain
57  ista integer start longitude on a task subdomain
58  iend integer end longitude on a task subdomain
59 
60  ista_2l integer start longitude -2 of the subdomain
61  iend_2u integer end longitude +2 of the subdomain
62  jsta_2l integer start latitude -2 of the subdomain
63  jend_2u integer end latitude +2 of the subdomain
64 
65 The shape of the subdomain is ista_2l:iend_2u,jsta_2l:jend_2u so it includes the halos although the halos are not populated until exhange is done in EXCH.f because of halos we need more bounds defined
